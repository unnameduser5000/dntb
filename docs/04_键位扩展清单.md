# DNTB 键位扩展清单

这份文档只回答一个具体问题：基于当前项目已经落地的 `KeyProgram -> ActionPlan -> ActionTrace -> WeaponCombo` 架构，接下来有哪些“新的移动方式”和“新的武器玩法”最适合接进键位系统，以及每一类改动落地时最少要补哪些点。

## 1. 当前事实边界

先把现在已经被代码确认的边界写清楚，避免后面 checklist 混淆“可装进键位的 token”和“武器派生技”：

- 当前键位系统真实数据源是 `scripts/core/ActionProgramController.gd`。
- 当前合法可编程 token 只有 `U / D / L / R / F / TL / TR`。
- `DirectionalTechniqueResolver.gd` 只会把这些 token 翻译成基础 action，不会直接把某个武器技写进键位计划。
- 当前武器技触发依赖 `ActionTrace`，不是依赖键位里直接放一个 `lunge` 或 `sweep`。
- `lunge`、`sweep` 已经是可执行 `ActionDef`，但它们是 follow-up action，不是自然掉落到键位池里的 token。
- `jump`、`move_back` 这两个动作资源已经存在，运行时和预览层也有实现，但现在还没有开放成可编程 token。
- 当前奖励逻辑还没有正式发放“新键位 token”；`RewardDef.gd` 有 `add_key` 骨架，但 `Game.gd` 的实际奖励表暂时只发 modifier、属性和回复。

结论：

- “把移动方式装进键位”是现有主路径。
- “把武器技直接装进键位”不是现有主路径，原则上不建议优先这么做。
- 更合适的做法是：把基础移动/转向 token 装进键位，再让武器从 `ActionTrace` 派生自己的技。

## 2. 已经可直接支撑的键位扩展

这部分优先列“代码已经基本准备好，只差开放入口”的项目。

### 2.1 后撤 token

推荐 token：`B`

为什么优先：

- `data/actions/move_back.tres` 已存在。
- `ActionResolver.gd` 已支持 `move_back` 的方向解析。
- `ActionTraceRecorder.gd` 已把它记为相对语义 `B`。
- `TurnController.gd` 已把它纳入链方向计算。

适合的玩法：

- 退后拉扯。
- 先后撤再前冲。
- 作为长枪、枪盾、陷阱型武器的起手语义。

Checklist：

- [ ] 在 `ActionProgramController.gd` 增加 `B` 的显示名、合法 token 集和掉落池入口。
- [ ] 在 `DirectionalTechniqueResolver.gd` 把 `B` 映射到 `move_back`。
- [ ] 在背包/键位 UI 中显示 `B` 的中文名和预览信息。
- [ ] 明确 `B` 是初始就给、战后奖励给，还是只在特定房间解锁。
- [ ] 给 `SmokeTest.gd` 补至少 3 个点：build plan、trace 记录为 `B`、预览方向正确。

### 2.2 跳跃 token

推荐 token：`J`

为什么优先：

- `data/actions/jump.tres` 已存在，基础参数已经定义为 `range = 2`。
- `ActionResolver.gd` 已有 `_resolve_jump()`。
- `ActionPreviewService.gd` 已有 `_preview_jump()`。
- `TurnController.gd` 已明确把 jump 排除在 momentum 叠加之外。

适合的玩法：

- 越过一格空地或威胁格。
- 作为重武器蓄势位移。
- 给 world slice 和战斗房间都增加更明显的路径选择。

Checklist：

- [ ] 在 `ActionProgramController.gd` 增加 `J` 的显示名、合法 token 集和掉落池入口。
- [ ] 在 `DirectionalTechniqueResolver.gd` 把 `J` 映射到 `jump`。
- [ ] 决定 `jump` 是否允许放进 starter preset，还是作为中期解锁 token。
- [ ] 确认 `jump` 是否允许与某些武器碰撞效果联动；当前默认不会参与冲撞速度叠层。
- [ ] 给 `SmokeTest.gd` 补至少 4 个点：plan build、预览落点、落点被挡住时失败、不会累积 momentum。

### 2.3 侧移 token

推荐 token：`SL` / `SR`

为什么是中优先级：

- `ActionTrace` 已经天然使用 `SL / SR` 描述相对侧移语义。
- 但当前没有对应的基础 `ActionDef` 资源，也没有直接把“相对方向移动”翻译成 action 的 resolver 分支。

适合的玩法：

- 匕首、双刀、拳套这类围绕绕背和侧切的武器。
- 强调“朝向管理”的移动博弈。

Checklist：

- [ ] 新建 `step_left`、`step_right` 的 `ActionDef` 资源。
- [ ] 约定它们是基于当前朝向做相对移动，而不是世界绝对方向。
- [ ] 在 `DirectionalTechniqueResolver.gd` 新增相对 token 到 action 的映射。
- [ ] 在 `ActionPreviewService.gd` 和 `ActionResolver.gd` 统一相对侧移的方向计算。
- [ ] 给 `SmokeTest.gd` 补侧移成功、侧移被挡、trace 仍正确记录为 `SL/SR`。

### 2.4 冲刺 token

推荐 token：`Dash`

为什么是中优先级：

- 当前系统已经支持 `range > 1` 的移动 action。
- 但冲刺会和 `ImpactShield` 的碰撞、chain speed、预览穿格规则发生明显交互，不能只做资源配置。

适合的玩法：

- 重盾、战锤、骑枪一类“用位移换输出”的武器。
- 让玩家在单键里写更强的爆发位移，而不必总靠 `F,F`。

Checklist：

- [ ] 新建 `dash` 的 `ActionDef`，明确 `range`、是否停在首个敌人格、是否可穿过陷阱。
- [ ] 明确 dash 是否计入 momentum；建议先不叠、或单独定义。
- [ ] 统一预览层与运行时对“撞到敌人/墙时停在哪里”的规则。
- [ ] 为 `ImpactShieldDef` 明确 dash 的碰撞伤害口径，避免和 `F,F` 重复叠收益。
- [ ] 补至少 4 个烟测：空地冲刺、撞墙、撞敌、与冲击盾联动。

## 3. 最适合现架构的新武器方向

这部分说的是“武器如何借现有键位语义和 `ActionTrace` 吃到新玩法”，不是说把整把武器当 token 拖进键槽。

### 3.1 长枪 / 刺枪

匹配度：高

原因：

- `WeaponTechniqueDef.PatternType.SAME_MOVE_DIRECTION` 已经存在。
- 现有 `impact_lunge.tres` 就是同方向连续移动两次的成功示例。

适合的触发：

- `F -> F` 触发短突刺。
- `F -> F -> F` 触发长突刺或贯穿。
- `B -> F` 触发回马枪。

Checklist：

- [ ] 新建武器资源，例如 `spear.tres`。
- [ ] 复用或扩展 `lunge`，再补一个更长距离的 follow-up action。
- [ ] 新增 `WeaponTechniqueDef`，优先用 `SAME_MOVE_DIRECTION` 和 `SYMBOL_SEQUENCE` 混合描述。
- [ ] 先只做直线攻击，不要第一版就引入穿透、拖拽、出血三套附加规则。

### 3.2 巨剑 / 斧戟

匹配度：高

原因：

- 当前 `sweep` 已验证 `TL -> TR` 的 sequence pattern 可用。
- 这类武器最容易利用转向和扇形攻击，而不要求复杂弹道。

适合的触发：

- `TL -> TR` 触发横扫。
- `TR -> TL` 触发反向横扫。
- `TL -> TL -> F` 触发回旋斩后前压。

Checklist：

- [ ] 保留“基础按键写的是转向和移动，不是直接写横扫”的原则。
- [ ] 新建至少 2 个 `WeaponTechniqueDef`，分别覆盖左右向序列。
- [ ] 先统一横扫的攻击格模板，再决定是否加击退。
- [ ] 若要加击退，优先走 `EffectPipeline`，不要把额外规则塞进 UI。

### 3.3 匕首 / 双刀

匹配度：中高

原因：

- 它最适合吃 `SL / SR / B` 这些相对移动语义。
- 当前 trace 层已经能表达这些语义，只差把它们开放为更好用的输入或形成稳定 pattern。

适合的触发：

- `SL -> SR` 触发交叉切。
- `B -> F` 触发穿步返刺。
- `SL -> F` 或 `SR -> F` 触发绕身突刺。

Checklist：

- [ ] 先决定是否先做 `B`，再决定要不要同步做 `SL/SR`。
- [ ] 第一版优先做高机动、低伤害、多次触发，不要做复杂中毒叠层。
- [ ] 若涉及“绕背加伤”，把判定写在武器技 follow-up 或 modifier，不写在键位编辑层。

### 3.4 盾锤 / 重盾

匹配度：高

原因：

- `resolve_move_collision()` 已经是现成扩展点。
- 当前 `ImpactShieldDef` 已把“连续移动换冲撞收益”跑通。

适合的触发：

- `F -> F` 增强冲撞。
- `J -> F` 做跳砸或落地震荡。
- `G -> F` 做防守反击式盾击。

Checklist：

- [ ] 若继续扩冲撞流，优先区分“地面冲撞”和“跳跃落地”两套收益。
- [ ] 新增武器时尽量复用 `resolve_move_collision()`，不要复制一份完整移动逻辑。
- [ ] 如果加入震荡、击飞、落地波纹，优先封装为新的 effect packet 或 modifier 反应。

### 3.5 战锤 / 重击武器

匹配度：中

原因：

- 攻击和 guard 资源都已存在，但“重击蓄力”目前没有单独 token 和状态层。
- 适合做低频高伤的 follow-up 技，而不是先做复杂蓄力 UI。

适合的触发：

- `G -> F` 触发蓄势重击。
- `B -> B -> F` 触发后撤蓄力猛冲。

Checklist：

- [ ] 第一版先做 pattern 触发重击，不做跨回合蓄力。
- [ ] 若重击要附带眩晕或破甲，优先走 `EffectPipeline`。
- [ ] 先确认敌人身上是否已有足够状态承载点；没有就只做伤害和击退。

## 4. 当前不建议优先的方向

### 4.1 远程弓 / 枪 / 法杖弹道武器

暂缓原因：

- 当前 `ActionResolver` 的攻击主要是近战格判定。
- 没有成熟的弹道、飞行物、命中路径阻挡、落点效果这一整套边界。
- 如果硬塞进现有键位系统，会先把 resolver 和表现层复杂度拉高。

Checklist：

- [ ] 先决定是否需要 projectile 层。
- [ ] 先决定远程攻击是否要支持穿墙、穿怪、落地范围伤害。
- [ ] 没有统一弹道模型前，不建议开做具体武器资源。

### 4.2 直接把武器技 token 化

暂缓原因：

- 这会绕开当前 `ActionTrace -> WeaponCombo` 的边界。
- 会把“基础输入”和“武器收益”重新耦合回 `KeyProgram`。
- 现有 smoke test 已明确验证 `lunge`、`sweep` 不进入 key token pool。

Checklist：

- [ ] 除非设计目标明确改成“技能宏”游戏，否则不要把 `lunge`、`sweep` 直接变成可拖拽 token。
- [ ] 新武器优先新增 trace pattern 和 follow-up action，而不是新增直接可拖拽武器技 token。

## 5. 推荐落地顺序

如果只按“投入产出比”排序，建议顺序如下：

1. `B` 后撤 token。
2. `J` 跳跃 token。
3. 基于 `B/F` 与 `F/F/F` 的长枪或枪盾武器。
4. 基于 `TL/TR` 的第二把横扫系武器。
5. `SL/SR` 侧移 token。
6. 基于侧移语义的匕首/双刀武器。
7. `Dash` 或更复杂的位移 token。

## 6. 每次新增键位玩法的通用验收清单

无论新增的是移动 token 还是武器技，至少都要检查下面这些点：

- [ ] `ActionProgramController.gd` 已接受新 token，存档读写不丢。
- [ ] `DirectionalTechniqueResolver.gd` 能把新 token 翻译成基础 action。
- [ ] `ActionResolver.gd` 与 `ActionPreviewService.gd` 对方向和阻挡的规则一致。
- [ ] `ActionTraceRecorder.gd` 产出的 symbol 与设计预期一致。
- [ ] `TurnController.gd` 对 chain speed / momentum 的处理已明确，不靠默认碰运气。
- [ ] `BattleUI` 与背包页能正确显示 token 名称、预览和拖拽结果。
- [ ] `SmokeTest.gd` 至少覆盖 plan、preview、runtime 三类结果。

## 7. 一句话决策建议

如果下一轮只做一小步，最值得先做的是：

- 开放 `B`
- 开放 `J`
- 用这两个 token 去养出第二把真正依赖键位编排差异的武器

这样改动最贴近现有架构，能最快把“键位编排真的改变战斗手感”这件事做得更明显。
