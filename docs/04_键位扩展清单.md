# DNTB 键位扩展清单

这份文档只回答一个问题：基于当前已经落地的 `KeyProgram -> DirectionalTechniqueResolver -> ActionResolver` 主路径，下一批最适合接进键位系统的“新移动方式”和“新武器”是什么，以及每类改动落地时最少要补哪些点。

## 1. 当前事实边界

先把现状写清楚，避免再把旧武器技模型混回键位系统：

- 当前键位系统真实数据源是 `scripts/core/ActionProgramController.gd`。
- 当前合法可编程 token 包括 `U / D / L / R / F / B / TL / TR / A / G / W / J`。
- `DirectionalTechniqueResolver.gd` 负责把 token 翻译成运行时动作。
- `A` 是通用攻击 token，不直接绑定固定武器；它会在 build plan 时解析成 `active_weapon.attack_action`。
- 当前武器模型是“一把武器对应一个攻击动作”，不是“一把武器挂一组 combo/hook”。
- `ActionTrace` 仍然保留，但当前主要用于记录执行语义、调试和后续输入分析，不再承担武器主触发链。
- 当前奖励逻辑还没有正式发放大量新 token，主路径仍以默认键位和未分配 token 池为主。

结论：

- 现在的键位系统已经是“基础行动编排器”，不只是移动宏。
- 新武器优先新增 `ActionDef`，然后由武器资源声明 `attack_action`。
- 不再推荐把武器特性建成独立 `WeaponTechniqueDef` 或命中 hook。

## 2. 当前已经接入的基础行动 token

这部分是已经走通 `ActionProgramController -> DirectionalTechniqueResolver -> ActionResolver` 的内容：

- 绝对方向移动：`U / D / L / R`
- 相对基础移动：`F / B`
- 朝向控制：`TL / TR`
- 战斗基础动作：`A / G / W`
- 位移动作：`J`

这意味着一个键位链里已经可以混排：

- 移动
- 转向
- 攻击
- 防御
- 等待
- 跳跃

## 3. 已经可直接支撑的键位扩展

这部分优先列“代码结构已经基本准备好，只差开放入口”的项目。

### 3.1 侧移 token

推荐 token：`SL` / `SR`

为什么优先：

- 当前基础动作层已经覆盖前后、转向、攻击、防御、等待，最自然的缺口就是相对横移。
- `ActionTrace` 已经天然使用 `SL / SR` 描述相对侧移语义。
- 匕首、抓取、绕背、瞬步这类玩法都更依赖侧移，而不是再叠一层旧式武器判招。

Checklist：

- [ ] 新建 `step_left`、`step_right` 的 `ActionDef` 资源。
- [ ] 在 `DirectionalTechniqueResolver.gd` 增加 `SL / SR` 到动作的映射。
- [ ] 在 `ActionResolver.gd` 和 `ActionPreviewService.gd` 统一相对方向换算。
- [ ] 明确侧移是否改变面朝方向；建议第一版不改面朝。
- [ ] 校对 `ActionTraceRecorder.gd` 在成功侧移和侧移失败时的记录口径。

### 3.2 冲刺 token

推荐 token：`Dash`

为什么优先：

- 当前系统已经支持 `range > 1` 的移动动作。
- `MovementResult` 已经能承载多步位移、阻挡和停点。
- 它能直接和击退、换位、拉拽、跳跃这些效果形成联动。

Checklist：

- [ ] 新建 `dash` 的 `ActionDef`。
- [ ] 明确 dash 使用当前朝向还是显式方向；建议第一版用当前朝向。
- [ ] 明确撞墙、撞人、越障时的停点规则。
- [ ] 在 `ActionPreviewService.gd` 和 `ActionResolver.gd` 复用同一套停点口径。
- [ ] 明确 dash 是否计入 momentum；建议与普通 `F` 分开。

### 3.3 抓取/投技前置 token

推荐 token：`Grab` 或 `ThrowPrep`

为什么优先：

- 用户已经明确问过背摔这类攻击方式。
- 这类内容会直接验证“攻击动作 + 移动 API + 表现事件”三层边界是否够稳。
- 先把它做成动作层，比恢复武器 hook 更干净。

Checklist：

- [ ] 明确它是单步攻击动作，还是先抓再摔的两段动作。
- [ ] 明确失败口径：目标不存在、目标友军、目标背后落点被挡住时分别怎么处理。
- [ ] 明确是否需要新 effect packet，例如 `throw`，还是先由 `damage + move/swap` 组合。
- [ ] 明确是否需要新 presentation frame，例如 `throw_start`、`throw_land`。

## 4. 最适合当前架构的新武器方向

这里说的是“武器如何声明自己的攻击动作”，不是把整把武器当 token 拖进键槽。

### 4.1 长枪 / 刺枪

匹配度：高

原因：

- 现有动作层已经有 `charge_thrust`。
- 直线攻击、较长攻击距离、前压站位都和当前动作模型天然兼容。

Checklist：

- [ ] 武器资源声明 `attack_action = charge_thrust`。
- [ ] 明确 `charge_thrust` 的攻击距离、是否前移、是否允许穿透。
- [ ] 若增加命中后前压，优先走动作内部调用移动 API。
- [ ] 第一版只做直线差异，不要同时上出血、拖拽、破甲。

### 4.2 巨剑 / 斧戟

匹配度：高

原因：

- 现有动作层已经有 `great_sweep`。
- 扇形攻击比旧 combo 模型更适合直接表达为一个攻击动作。

Checklist：

- [ ] 武器资源声明 `attack_action = great_sweep`。
- [ ] 统一横扫的攻击格模板。
- [ ] 若加击退，优先走 `EffectPipeline` 或动作内部调用击退 helper。
- [ ] 不要把左右横扫拆回两套武器脚本逻辑。

### 4.3 盾锤 / 重盾

匹配度：高

原因：

- 盾类武器最适合验证“攻击动作”和“移动动作”之间的联动，而不是命中 hook。
- 它可以自然扩成盾击、冲撞、跳砸、守势反击。

Checklist：

- [ ] 第一版先选一个主攻击动作，例如近身盾击。
- [ ] 若要扩冲撞收益，优先通过新动作调用 `MovementResult` 路径，不恢复旧碰撞武器 API。
- [ ] 若要做 `G -> F` 风格反击，优先建成单独攻击动作，不写成武器被动脚本。

### 4.4 匕首 / 双刀

匹配度：中高

原因：

- 它最适合吃 `SL / SR / B` 这些相对位移输入。
- 只要侧移 token 开放，匕首就能很快做出和长枪、巨剑不同的节奏。

Checklist：

- [ ] 先补 `SL / SR`，再做匕首。
- [ ] 第一版优先做高机动、低伤害。
- [ ] 若涉及绕背加伤，优先把判定写进攻击动作，不写进键位编辑层。

### 4.5 背摔 / 投技武器

匹配度：中高

原因：

- 能直接检验当前动作 API 是否足够表达“命中 + 位移 + 落点失败 fallback”。
- 这也是最能暴露移动结果层缺口的一类武器。

Checklist：

- [ ] 新建 `back_suplex` 或同类攻击动作。
- [ ] 明确触发方向、目标落点、是否换位。
- [ ] 明确落点阻挡时是失败、原地伤害，还是转成撞墙收益。
- [ ] 为它补独立表现事件，不只复用普通 hit。

## 5. 当前不建议优先的方向

### 5.1 远程弹道武器

暂缓原因：

- 当前攻击主路径仍偏近战格判定。
- 还没有成熟的 projectile、路径阻挡、飞行表现协议。
- 现在硬做会先把 resolver 和表现层复杂度一起拉高。

Checklist：

- [ ] 先决定是否需要 projectile 层。
- [ ] 先定义穿墙、穿怪、落点爆炸等规则边界。
- [ ] 没有统一弹道模型前，不建议开做具体弓/枪/法杖资源。

### 5.2 把武器特性重新做成独立 hook 系统

暂缓原因：

- 这会重新引入第二套战斗规则入口。
- 现在主路径已经是“武器声明攻击动作”，继续扩动作 API 更稳定。
- 同一类差异若分散到武器脚本里，预览、效果和表现都会再次分叉。

Checklist：

- [ ] 新武器优先新增 `ActionDef`。
- [ ] 新攻击收益优先新增动作内部规则或 effect packet。
- [ ] 只有在动作 API 明显无法表达时，再讨论是否需要新增更通用的战斗事件接口。

## 6. 推荐落地顺序

如果只按投入产出比排序，建议顺序如下：

1. `SL / SR` 侧移 token。
2. `Dash` 或其他高承诺位移 token。
3. 基于动作 API 的长枪/枪盾扩展。
4. 基于扇形攻击的第二把重武器。
5. 基于侧移语义的匕首/双刀武器。
6. 背摔、抓取、投技类动作武器。

## 7. 每次新增键位玩法的通用验收清单

无论新增的是移动 token 还是武器攻击动作，至少都要检查下面这些点：

- [ ] `ActionProgramController.gd` 已接受新 token，存档读写不丢。
- [ ] `DirectionalTechniqueResolver.gd` 能把新 token 翻译成正确动作。
- [ ] `ActionResolver.gd` 与 `ActionPreviewService.gd` 对方向、阻挡和停点规则一致。
- [ ] 新动作如果带位移，必须走现有移动 API，而不是直接手写 `grid.move_actor()`。
- [ ] 新动作如果带追加伤害/击退/换位，优先走 `EffectPipeline` 或 resolver helper。
- [ ] `ActionTraceRecorder.gd` 产出的 symbol 与设计预期一致。
- [ ] `BattleUI` 与背包页能正确显示 token 名称、预览和拖拽结果。
- [ ] 文档说明的是当前真实主路径，而不是旧 combo/hook 模型。

## 8. 一句话决策建议

如果下一轮只做一小步，最值得先做的是：

- 开放 `SL / SR`
- 再补一个高承诺位移键，例如 `Dash`
- 用这些更鲜明的移动语义去养出下一把依赖站位和节奏差异的武器

这样最贴近现在的主架构，也最能把“攻击和移动都被编入同一套键位输入系统”这件事做得更明显。
