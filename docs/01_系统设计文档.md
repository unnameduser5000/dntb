# DNTB 系统设计文档

## 1. 项目定位

DNTB 当前是一个小体量、可快速迭代的 Godot 4 战术 roguelite 原型。它的核心不是传统“选择技能卡再行动”，而是把键盘输入抽象成可编程按键槽：玩家在安全节点、休息点或酒馆休息区调整 `QWER / ASDF / ZXCV` 十二个物理键位上的行动 token，进入战斗或离开安全区后按键槽锁定，按下某个键位时会按槽内从左到右的顺序依次执行绑定内容。

当前已经落地的体验包括：

- 8x8 网格房间、墙体、玩家、敌人和钥匙拾取。
- 营地/休息点可调整行动 token，战斗中锁定。
- 世界切片里只有站在酒馆休息区时才可调整按键编排。
- 战斗内执行玩家行动链，然后敌人根据意图行动。
- 房间胜利后给奖励，奖励可增加行动、遗物或属性。
- 冲击盾武器、连续移动速度、撞击伤害和击退。
- EffectPipeline 支持伤害、移动、击退、事件反应和遗物式修饰器。
- 成就、设置、存档、输入绑定、随机服务等基础 autoload。

## 2. 运行入口

项目主场景在 `project.godot` 中配置为：

```text
res://scenes/game/App.tscn
```

入口结构分两层：

- `App.tscn`：应用外壳，负责主菜单、设置菜单、暂停菜单和游戏可见性切换。
- `Game.tscn`：核心玩法场景，负责棋盘、战斗 UI、回合控制、行动解析、敌人规划和 run 控制。

`App.tscn` 当前节点职责：

```text
App
├── AppBackground
├── Game
└── MenuLayer
    ├── MainMenu
    ├── SettingsMenu
    └── PauseMenu
```

`Game.tscn` 当前节点职责：

```text
Game
├── BoardView          # 棋盘显示
├── ActorRoot          # 演员表现根节点
├── CanvasLayer
│   └── BattleUI       # 战斗 HUD、行动槽、奖励和结算覆盖层
├── TurnController     # 回合流程
├── ActionResolver     # 行动规则解析
├── EnemyPlanner       # 敌人行动和威胁预览
└── RunController      # 早期 run 控制骨架
```

## 3. 目录分层

当前项目采用 Godot 常见的“场景、脚本、数据、资源”分离结构：

- `scenes/game/`：入口和核心玩法场景。
- `scenes/map/`：棋盘视图和棋盘格。
- `scenes/ui/`：菜单、HUD、UI 组件和主题。
- `scenes/actors/`：演员表现层场景。
- `scripts/core/`：核心服务、控制器和规则协调层。
- `scripts/data/`：Godot Resource 数据定义，例如 ActionDef、ActorDef、WeaponDef、EffectModifierDef。
- `scripts/runtime/`：一局战斗中的运行态对象，例如 ActorState、ActionInstance、EffectPacket。
- `scripts/view/`：场景脚本和 UI 表现层。
- `data/`：实际资源实例，包括角色、行动、遗物、武器和成就。
- `addons/dialogue_manager/`：第三方 Dialogue Manager 插件，保留其原许可证和结构。

## 4. 核心数据流

一次玩家按键行动的大致路径如下：

```text
键盘事件
→ PlayerInputService 判断 player_key_q ... player_key_v
→ Game._submit_key_chain(key_id)
→ ActionProgramController 把按键槽 token 转成 ActionInstance 队列
→ TurnController.submit_player_plan(plan)
→ ActionResolver.resolve(action, state)
→ EffectPipeline 处理 EffectPacket、Modifier 和事件反应
→ GameState / GridModel 被更新
→ EnemyPlanner 预览或生成敌人行动
→ Game 刷新 BoardView 和 BattleUI
```

这个路径里最重要的边界是：

- `Game.gd` 现在仍是玩法总装层，负责 run 进度、房间生成、奖励应用、UI 信号连接和存档 provider 注册。
- `ActionProgramController.gd` 是按键槽真实数据源，当前管理的是 `QWER / ASDF / ZXCV` 十二个物理键位与未分配 token 池。
- `TurnController.gd` 只管回合阶段和行动执行顺序，不直接写具体战斗规则。
- `ActionResolver.gd` 管移动、攻击、转向、防御、武器 hook、死亡和胜负判断。
- `EffectPipeline.gd` 管效果包的修饰、执行和事件反应，适合承载遗物、武器、状态效果等可组合规则。

## 5. 回合与 run 流程

当前 run 路线还写在 `scripts/view/Game.gd` 的 `MAP_NODES` 和 `ROOMS` 常量中。流程大致是：

```text
营地休息点
→ 第一战
→ 战后休息点
→ 第二战
→ 回复休息点
→ Boss 战
→ 通关或失败
```

战斗内循环：

1. `TurnController.start_battle()` 进入 planning。
2. 玩家通过 UI 队列或按键槽提交行动链。
3. `TurnController` 顺序执行玩家行动，并为连续同方向移动计算 chain speed 和 momentum。
4. `ActionResolver` 解析每个行动，必要时交给武器和 EffectPipeline。
5. 若战斗未结束，`EnemyPlanner` 生成敌人计划并执行。
6. 清理临时 flag，回合数加一，重新进入 planning。

背包/编排页的使用规则：

- 可通过 `Tab` 或右下角 `背包 / Tab` 打开。
- 打开背包会暂停当前场景树，关闭后恢复。
- 左侧显示 `QWER / ASDF / ZXCV` 三行四列键位槽，以及永久增益列表。
- 右侧显示未分配 token；拖到左侧槽位后会追加到该键位链尾部。
- 永久增益当前通过 tooltip 提供详细说明。

## 6. 行动、武器和效果系统

### ActionDef

`ActionDef` 是行动资源，包含：

- `id`
- `display_name`
- `short_name`
- `kind`
- `range`
- `power`
- `direction_mode`

当前行动资源放在 `data/actions/`。其中移动、转向、攻击、等待、防御和方向移动 token 仍是基础行动资源；突刺、横扫虽然继续复用 `ActionDef` 资源，但当前语义已经调整为“由方向链派生出的武器技”，不再作为可拖拽的自然按键 token 进入行动编程层。

### WeaponDef

`WeaponDef` 是武器基类，提供这些可选 hook：

- `resolve_move_collision()`
- `resolve_attack_hit()`
- `resolve_attack_miss()`
- `resolve_action_chain_finished()`

当前 `impact_shield.tres` 使用 `ImpactShieldDef`，设计重点是“移动撞击”：连续同方向移动会提高速度，撞上敌人时按速度造成伤害和击退。

### TechniquePatternDef

`TechniquePatternDef` 是“方向链派生武器技”的数据定义层，用来把按键编程与武器技判定拆开。当前资源放在 `data/techniques/`，并由 `DirectionalTechniqueResolver.gd` 统一消费。

- `technique_id`
- `action`
- `match_kind`
- `consume_count`
- `priority`
- `direction_source`

当前示例：

- `lunge_pattern.tres`：两个相同方向 token 派生 `lunge`
- `sweep_pattern.tres`：一段包含 U/D/L/R 各一次的四向旋转派生 `sweep`

### ActionTrace

`ActionTrace` 是“动作执行结果记录层”，它不负责直接触发武器技，只负责把已经执行过的动作整理成更稳定、可被后续系统消费的语义轨迹。

当前实现重点：

- `ActionTraceEntry` 记录单步执行结果：
  - `actor_id`
  - `action_id`
  - `input_token_id`
  - `symbol`
  - `chain_index`
  - `from_cell` / `to_cell`
  - `facing_before` / `facing_after`
  - `tags`
- `TurnController.gd` 在每个 action 执行后记录 trace，因此这层不会改动现有战斗规则，只是把执行结果沉淀出来。
- 相对语义当前按“动作执行前朝向”记录；例如角色面朝右时执行 `U` 会记成 `SL`，随后若角色已转为面朝上再执行 `R`，会记成 `SR`。
- 当前 `symbol` 口径已经开始对齐团队讨论的“相对语义”：
  - `F`
  - `B`
  - `SL`
  - `SR`
  - `TL`
  - `TR`
  - 以及现有基础动作本身的 `combo_symbol`

这层的意义是把“输入 token”和“武器判招语义”之间先插入一层稳定边界。后面如果继续做：

- `WeaponPattern`
- `InputInterference`
- `Jump`
- 更复杂的相对动作识别

都可以优先建立在 `ActionTrace` 上，而不是继续直接读取 `ActionInstance` 或 `Game.gd` 的临时状态。

### WeaponTechniqueDef / WeaponComboResolver

`WeaponTechniqueDef` 和 `WeaponComboResolver` 是下一段架构的最小骨架：开始让武器从 `ActionTrace` 读取“最近发生过什么”，而不是继续只盯着绝对方向 token 链。

当前实现阶段：

- `WeaponTechniqueDef`
  - 描述一个武器技的 trace pattern
  - 当前使用 suffix exact match
- `WeaponComboResolver`
  - 读取 actor 的最近 `ActionTrace`
  - 读取武器上的 `combo_techniques`
  - 返回匹配到的 technique 列表

当前刻意保持为“只读识别层”：

- 会识别
- 会提供调试查询
- 还不直接改写现有战斗执行入口

这样做的目的是先把架构边界立起来，再逐步把旧的方向链派生逻辑迁走。

现在队友新增一个派生武器技，主路径已经变成：

1. 新建对应的 `ActionDef`
2. 新建一个 `TechniquePatternDef`
3. 在武器上声明 `weapon_technique_ids`
4. 再决定是否通过奖励、初始解锁或其他 run 事件把它开放给玩家

### Current combo architecture (2026-06-27 update)

The old `TechniquePatternDef` / `data/techniques/` path is no longer the live
combat entry point.

Current authoritative flow:

1. `KeyProgram` stores editable absolute input tokens such as `U/D/L/R`.
2. `DirectionalTechniqueResolver` only translates those tokens into base
   executable actions such as `move_key`.
3. Combat resolves the real action results first.
4. `ActionTraceRecorder` writes relative execution semantics such as
   `F/B/SL/SR/TL/TR`.
5. `WeaponComboResolver` matches `ActionTrace`, not `KeyProgram`.
6. `TurnController` can execute the matched weapon technique's follow-up
   `ActionDef`.

Important gameplay consequence:

- rest-site preview may predict likely combo hits from the planned key chain
- battle-time triggering is still based on real `ActionTrace`
- terrain, collisions, and effect-driven movement changes can therefore stop a
  combo from triggering even when the original key program looked correct

Current extension path for a new weapon technique:

1. add or reuse the follow-up `ActionDef`
2. add a `WeaponTechniqueDef`
3. wire it into the weapon's `combo_techniques`
4. decide how the run unlocks that technique id

### EffectPipeline

EffectPipeline 的扩展点是 `EffectModifierDef`：

- `modify_packets()`：在执行前修改、放大、复制或取消效果包。
- `react_to_event()`：对伤害、移动、击杀、击退等事件生成后续效果包。
- `priority`：控制修饰器处理顺序。
- `max_generation_depth` / `max_event_depth`：限制递归生成深度。

这套结构适合继续扩展遗物、诅咒、临时状态、武器词条和房间规则。

## 7. 视图与 UI

表现层主要在 `scripts/view/` 和 `scenes/ui/`：

- `BattleUI.gd`：战斗面板、行动队列、按键槽、敌人意图、奖励覆盖层、战斗结果。
- `BattleHud.gd`：玩家生命、SAN、房间、回合，以及玩家位置和朝向的统一状态框。
- `RunSidebar.gd`：右下角背包和菜单入口。`背包 / Tab` 会打开背包页并暂停当前场景；背包左侧显示 `QWER / ASDF / ZXCV` 三行四列的按键槽与永久增益，右侧显示未分配 token。菜单面板承接事件记录与调试状态。
- `BoardView.gd`：读取 `GameState` 渲染棋盘、危险格、预览移动格和攻击格。
- `UiKeySlot.gd` / `UiKeyToken.gd`：按键槽拖拽和 hover 预览。
- `SettingsMenu.gd`：分辨率、全屏、移动键重绑定。

UI 层的目标是展示和发信号，不直接拥有战斗规则。新增玩法规则时，优先放在 core、runtime 或 data 层，再由 view 层订阅状态刷新。

## 8. Autoload 服务

`project.godot` 当前注册了这些 autoload：

- `SettingsService`：窗口分辨率和全屏设置。
- `SaveService`：统一存档入口，收集 provider 的 `get_save_data()`。
- `SceneService`：场景切换、重载和退出。
- `AudioService`：音乐、音效和音频 bus 控制。
- `AchievementService`：成就定义、事件、进度和存档。
- `RandomService`：可保存/恢复的随机数服务。
- `PlayerInputService`：移动键 InputMap 和重绑定。
- `CurseService`：run 内诅咒计数和规则。
- `EnemySpawnService`：敌人定义注册和按楼层生成候选。
- `EconomyService`：经济相关服务骨架。
- `DialogueManager`：第三方 Dialogue Manager。
- `DialogueService`：项目对话服务封装。

新增全局服务前先确认是否真的需要 autoload。若只是某个场景内部可控的逻辑，优先用子节点、Resource 或 RefCounted 组合。

## 9. 当前设计债务

这些不是立即要修的 bug，而是后续演进时应优先处理的结构点：

- `Game.gd` 仍承担较多装配职责，房间、路线、奖励表还写在常量里。
- `RunController.gd` 已存在，但当前主要 run 流程还在 `Game.gd` 中。
- `RoomDef.gd` 已作为未来数据驱动房间的脚手架，尚未接管 `ROOMS`。
- UI 已较完整，但 `ActorRoot` 与演员表现层后续可继续拆清楚。
- 对话、经济、诅咒等服务已有入口，玩法接入程度还较轻。

推荐下一步拆分顺序：

1. 把 `ROOMS` 和 `MAP_NODES` 迁移为 Resource 数据或独立配置。
2. 让 `RunController` 接管路线推进、奖励构建和 run 持久状态。
3. 保持 `Game.gd` 为 composition root，只负责连接场景节点和协调刷新。
4. 为每个新增规则补一条 SmokeTest 或更小的 headless 测试。
