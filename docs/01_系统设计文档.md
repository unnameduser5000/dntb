# DNTB 系统设计文档

## 1. 项目定位

DNTB 当前是一个小体量、可快速迭代的 Godot 4 战术 roguelite 原型。它的核心不是传统“选择技能卡再行动”，而是把键盘输入抽象成可编程按键槽：玩家在安全节点、休息点或酒馆休息区调整 `QWER / ASDF / ZXCV` 十二个物理键位上的行动 token，进入战斗或离开安全区后按键槽锁定，按下某个键位时会按槽内从左到右的顺序依次执行绑定内容。

当前已经落地的体验包括：

- 8x8 网格房间、墙体、玩家、敌人和钥匙拾取。
- 营地/休息点可调整行动 token，战斗中锁定。
- 世界切片里只有站在酒馆安全屋 footprint 内的可走格上时才可调整按键编排；这包括室内地板、门口、院地以及 tavern 互动点。
- 世界切片酒馆安全区内会生成中立 NPC；靠近后可用确认键交谈，用轻量文本补充当前世界观。
- 战斗内执行玩家行动链，然后敌人根据意图行动。
- 房间胜利后给奖励，奖励可增加行动、遗物或属性。
- 冲击盾武器、连续移动速度、撞击伤害和击退。
- EffectPipeline 支持伤害、移动、击退、事件反应和遗物式修饰器。
- 成就、设置、存档、输入绑定、随机服务等基础 autoload。

## 2. 运行入口

项目主场景在 `project.godot` 中配置为：

```text
res://scenes/game/App.tscn
```

入口结构分两层：

- `App.tscn`：应用外壳，负责主菜单、设置菜单、暂停菜单和游戏可见性切换。
- `Game.tscn`：核心玩法场景，负责棋盘、战斗 UI、回合控制、行动解析、敌人规划和 run 控制。

`App.tscn` 当前节点职责：

```text
App
├── AppBackground
├── Game
└── MenuLayer
    ├── MainMenu
    ├── SettingsMenu
    └── PauseMenu
```

`Game.tscn` 当前节点职责：

```text
Game
├── BoardView          # 棋盘显示
├── ActorRoot          # 演员表现根节点
├── CanvasLayer
│   └── BattleUI       # 战斗 HUD、行动槽、奖励和结算覆盖层
├── TurnController     # 回合流程
├── ActionResolver     # 行动规则解析
├── EnemyPlanner       # 敌人行动和威胁预览
└── RunController      # 早期 run 控制骨架
```

## 3. 目录分层

当前项目采用 Godot 常见的“场景、脚本、数据、资源”分离结构：

- `scenes/game/`：入口和核心玩法场景。
- `scenes/map/`：棋盘视图和棋盘格。
- `scenes/ui/`：菜单、HUD、UI 组件和主题。
- `scenes/actors/`：演员表现层场景。
- `scripts/core/`：核心服务、控制器和规则协调层。
- `scripts/data/`：Godot Resource 数据定义，例如 ActionDef、ActorDef、WeaponDef、EffectModifierDef。
- `scripts/runtime/`：一局战斗中的运行态对象，例如 ActorState、ActionInstance、EffectPacket。
- `scripts/view/`：场景脚本和 UI 表现层。
- `data/`：实际资源实例，包括角色、行动、遗物、武器和成就。
- `addons/dialogue_manager/`：第三方 Dialogue Manager 插件，保留其原许可证和结构。

## 4. 核心数据流

一次玩家按键行动的大致路径如下：

```text
键盘事件
→ PlayerInputService 判断 player_key_q ... player_key_v
→ Game._submit_key_chain(key_id)
→ ActionProgramController 把按键槽 token 转成 ActionInstance 队列
→ TurnController.submit_player_plan(plan)
→ ActionResolver.resolve(action, state)
→ EffectPipeline 处理 EffectPacket、Modifier 和事件反应
→ GameState / GridModel 被更新
→ EnemyPlanner 预览或生成敌人行动
→ Game 刷新 BoardView 和 BattleUI
```

这个路径里最重要的边界是：

- `Game.gd` 现在仍是玩法总装层，负责 run 进度、房间生成、奖励应用、UI 信号连接和存档 provider 注册。
- `ActionProgramController.gd` 是按键槽真实数据源，当前管理的是 `QWER / ASDF / ZXCV` 十二个物理键位与未分配 token 池；池内 token 同时包含方向移动与基础动作输入。
- `TurnController.gd` 只管回合阶段和行动执行顺序，不直接写具体战斗规则。
- `ActionResolver.gd` 管移动、攻击、转向、防御、死亡和胜负判断；其中移动已经开始沉淀成独立 API，供效果层、动作层和后续预览层复用。
- `EffectPipeline.gd` 管效果包的修饰、执行和事件反应，适合承载遗物、武器、状态效果等可组合规则。

世界切片里的轻量 actor 交互当前走独立入口：

```text
确认键 / ui_accept / F
→ Game._submit_world_interact_action()
→ TurnController.submit_player_plan([interact])
→ ActionResolver._resolve_interact()
→ ActionResolver.world_npc_interaction_requested
→ Game._try_interact_with_world_npc()
→ ActorInteractionService 选择玩家面前一格的可交互 actor
→ 读取 ActorDef 上的交互文案
→ BattleUI 中下方交互框显示说话人和文本
→ 交互框存在时 Game 冻结世界切片常规输入与回合提交
→ GameState.messages / 可选 DialogueService
→ Game 刷新 BoardView 和 BattleUI
```

当前统一边界：

- `ActorDef` 同时承载基础战斗属性与交互能力字段，例如 `interaction_enabled`、`interaction_title`、`interaction_lines`。
- `NpcDef` 现在只是 `ActorDef` 的轻量子类，不再维护另一套独立交互字段。
- 怪物和 NPC 都是 `ActorState`；因此怪物可以被交互，NPC 也可以被攻击、受伤和死亡。

当前初始 NPC 是 tavern `npc_spawn_slots` 生成的 `tavern_keeper`：

- 地图字形是 `K`。
- 目前以文字 glyph 方式显示，颜色来自 `tavern_keeper.tres` 的 actor color。
- 玩家走到其相邻格、并面向目标后，可按确认键或物理 `F` 键槽开始交谈。
- 底层已接入一个专门的 `interact` ActionDef，世界切片交谈不再绕过动作系统直接触发。
- 交谈内容会出现在屏幕中下方对话框；每次只显示一条，按任意键关闭，不再在对话框内翻页。
- 对话框存在时，游戏不提交新的玩家行动，因此敌人的下一轮执行也不会在交谈结束前开始。
- 出生 tavern 的玩家点和 `酒馆掌柜` 点位当前都走固定引导布局：玩家落在室内固定出生格，掌柜落在相邻固定格，并且开局会自动面向掌柜，确保第一次对话和攻击 token 教学稳定可达。

当前世界切片怪物生成逻辑：

- 首批怪物由 `WorldSliceController._pick_enemy_spawn_cells()` 在建图完成后一次性补齐。
- 选位时会避开玩家出生点与已保留格，优先离出生区更远、与已选敌人有一定间距、并靠近挑战入口或废墟等高压位置。
- 世界运行中还会由 `refresh_streamed_enemies()` 做动态补怪与清怪：
  - 距离玩家超过 `STREAM_DESPAWN_DISTANCE` 的敌人会被回收；
  - 当活动敌人数低于 `STREAM_DESIRED_ACTIVE_ENEMY_COUNT` 时，会在玩家外圈半径带内重新补怪；
  - 补怪范围当前由 `STREAM_SPAWN_RADIUS_MIN` 到 `STREAM_SPAWN_RADIUS_MAX` 控制。

世界切片里的 NPC 生成入口当前是显式 API，而不是散落在多个 view 脚本里：

```text
BuildingPatternLibrary.tavern pattern
→ npc_spawn_slots
→ BuildingPlacementService._build_poi_record()
→ MapData.register_poi_record()
→ WorldSliceController._spawn_safe_zone_npcs()
→ WorldSliceController._pick_safe_zone_npc_cell_for_slot()
→ GameState.world_actor_positions / tracked_world_actor_id
```

当前第一版约束：

- 只有 tavern pattern 声明了 `npc_spawn_slots`。
- 只有玩家出生所在的 tavern footprint 会参与本轮 NPC 生成。
- 默认只生成一位 `tavern_keeper`，并且会自动成为 tracked actor；旧 `npc` 命名字段当前仅保留兼容镜像。

当前世界切片 POI 指示与遗迹交互边界：

- 当前 sidebar/debug 指示系统会额外维护两条 POI 方向提示：
  - `Boss遗迹`：当前指向 `challenge_entrance`
  - `最近小遗迹`：当前指向离玩家最近的 `ruin`
- 这两条提示当前都走 world slice 既有的相对方向文案口径，例如 `北（12 格）`。
- 第一版小遗迹不走 actor 交互，而是走“站到 ruin 的 `interaction_cell` 上按确认键调查”的 POI 交互。
- 当前 ruin 调查奖励第一版先收敛为固定教学收益：把 `SL / SR` 放进未分配 token 池。
- 每处 ruin 只可调查一次；已调查的小遗迹会提示已经搜索过，不会重复发放奖励。

当前怪物内容边界：

- 战斗房间内的正式敌人资源当前主要是 `slime`、`brute`、`boss` 三类。
- 世界切片里活动敌人本质上仍复用同一套 `ActorDef -> ActorState -> EnemyPlanner -> ActionResolver` 主链，只是出生入口不同。
- 当前怪物差异主要还集中在：
  - 基础数值
  - 站位压力
  - 近战攻击方式
  - 世界切片中的刷新位置与密度
- 这意味着“下一步丰富怪物内容”不该优先继续堆更多同质近战皮肤，而应优先补：
  - 更清晰的敌人战术角色
  - 更稳定的敌人意图模板
  - 能迫使玩家重新编排 token 的战场压力

推荐的怪物扩展顺序：

1. 先补一只会明显改变玩家站位决策的敌人，例如长距离直线威胁型或横扫压位型。
2. 再补一只会制造位移或干扰的敌人，例如击退、拉拽或换位型。
3. 然后补一只防守/蓄力型敌人，用来验证等待、防御、绕背和 burst 时机。
4. 最后再考虑把世界切片特殊敌人和房间专属敌人分层，而不是一开始就扩大量敌人资源。

## 5. 回合与 run 流程

当前 run 路线还写在 `scripts/view/Game.gd` 的 `MAP_NODES` 和 `ROOMS` 常量中。流程大致是：

```text
营地休息点
→ 第一战
→ 战后休息点
→ 第二战
→ 回复休息点
→ Boss 战
→ 通关或失败
```

战斗内循环：

1. `TurnController.start_battle()` 进入 planning。
2. 玩家通过 UI 队列或按键槽提交行动链。
3. `TurnController` 顺序执行玩家行动，并为连续同方向移动计算 chain speed 和 momentum。
4. `ActionResolver` 解析每个行动，必要时交给武器和 EffectPipeline。
5. 若战斗未结束，`EnemyPlanner` 生成敌人计划并执行。
6. 清理临时 flag，回合数加一，重新进入 planning。

背包/编排页的使用规则：

- 可通过 `Tab` 或右下角 `背包 / Tab` 打开。
- 打开背包会暂停当前场景树，关闭后恢复。
- 左侧显示 `QWER / ASDF / ZXCV` 三行四列键位槽，以及固定高度的永久增益列表。
- 每个键位卡片内部按固定槽位格子显示可放入的 token 数量；格子数量和每行列数当前由 `BagUI.gd` 参数控制。
- 右侧显示未分配 token 的固定网格背包；默认 5 列多行，列数和展示容量当前由 `BagUI.gd` 参数控制。
- 拖到左侧槽位后会追加到该键位链尾部。
- 永久增益当前通过 tooltip 提供详细说明；动作 token 的 hover 简介目前走 UI 本地说明兜底，尚未接到统一动作描述 API。

当前 token 获取方式：

- 开局 starter preset 会直接把一组基础 token 写进物理键槽；这部分不进入 pool。
- 地图上的 token 走自动拾取，玩家踩上去后会进入未分配 token 池。
- 出生 tavern 的 `酒馆掌柜` 首次对话会把通用攻击 token `A` 放进未分配 token 池，并明确提示玩家自己去分配键位。
- 当前战后奖励还不会正式发放大量新 token；主路径仍以默认槽位、地图拾取和 tavern 首次引导奖励为主。

推荐的内容获取分工：

- 动作 / token：以地图掉落为主，辅以少量 NPC 首次教学奖励。
- 稀有 token 或成套 token：更适合放进宝箱、神龛或特殊世界切片 POI 奖励。
- 武器：以遗迹奖励、宝箱奖励、战后奖励或关键 NPC 奖励为主，不建议主要由普通怪物路边掉落。
- 这样分工的原因是：
  - token 粒度小、组合强，适合频繁进入 pool 供玩家重新编排；
  - 武器粒度大、决定 build 方向，更适合作为节点级或房间级选择奖励。
- 因此后续第二批、第三批内容的推荐落地方式应是：
  - 新 token 先通过 tavern / 训练 NPC / 小型遗迹做首次教学；
  - 稳定后再进地图掉落池；
  - 新武器优先通过遗迹、宝箱或战后二选一奖励发放。
- 当前第一批已经开始按这条分工落地：
  - `A` 由 tavern 掌柜首次对话教学；
  - `SL / SR` 由小遗迹调查教学；
  - `twin_daggers` 仍保留为更适合挂到遗迹/宝箱/战后奖励的风格武器，而不是普通地图掉落。

当前 token 设计边界：

- token 代表“基础输入语义”，不是整把武器或整段连招。
- 当前合法可编程 token 包括：
  - 方向：`U / D / L / R`
  - 相对动作：`F / B / SL / SR / TL / TR`
  - 基础战斗/通用动作：`A / I / G / W / J`
- `A` 是通用攻击 token；真正打出的攻击动作由当前 `active_weapon.attack_action` 决定。
- 当前只有“移动到槽位”和“拖回 pool”，还没有正式的丢弃 token 入口。

## 6. 行动、武器和效果系统

### ActionDef

`ActionDef` 是行动资源，包含：

- `id`
- `display_name`
- `short_name`
- `kind`
- `range`
- `power`
- `direction_mode`

当前行动资源放在 `data/actions/`。其中方向移动、前进、后退、转向、攻击、等待、防御、跳跃都属于可进入按键编程层的基础行动资源；突刺、横扫虽然继续复用 `ActionDef` 资源，但当前语义已经调整为“由基础行动执行结果派生出的武器技”，不再作为可拖拽的直接键位 token。

### WeaponDef

`WeaponDef` 现在是轻量数据资源，不再承担独立战斗 hook。当前模型是：

- 一把武器对应一个 `attack_action`
- 玩家按下通用攻击 token 时，会在计划构建阶段解析成当前武器声明的具体攻击动作
- 武器差异主要通过不同 `ActionDef` 体现，而不是通过武器侧脚本直接改写战斗规则

当前示例：

- `impact_shield` -> `attack`
- `iron_spear` -> `charge_thrust`
- `greatblade` -> `great_sweep`
- `twin_daggers` -> `attack`

### TechniquePatternDef

`TechniquePatternDef` 是“方向链派生武器技”的数据定义层，用来把按键编程与武器技判定拆开。当前资源放在 `data/techniques/`，并由 `DirectionalTechniqueResolver.gd` 统一消费。

- `technique_id`
- `action`
- `match_kind`
- `consume_count`
- `priority`
- `direction_source`

当前示例：

- `lunge_pattern.tres`：两个相同方向 token 派生 `lunge`
- `sweep_pattern.tres`：一段包含 U/D/L/R 各一次的四向旋转派生 `sweep`

### ActionTrace

`ActionTrace` 是“动作执行结果记录层”，它不负责直接触发武器技，只负责把已经执行过的动作整理成更稳定、可被后续系统消费的语义轨迹。

当前实现重点：

- `ActionTraceEntry` 记录单步执行结果：
  - `actor_id`
  - `action_id`
  - `input_token_id`
  - `symbol`
  - `chain_index`
  - `from_cell` / `to_cell`
  - `facing_before` / `facing_after`
  - `tags`
- `TurnController.gd` 在每个 action 执行后记录 trace，因此这层不会改动现有战斗规则，只是把执行结果沉淀出来。
- 相对语义当前按“动作执行前朝向”记录；例如角色面朝右时执行 `U` 会记成 `SL`，随后若角色已转为面朝上再执行 `R`，会记成 `SR`。
- 当前 `symbol` 口径已经开始对齐团队讨论的“相对语义”：
  - `F`
  - `B`
  - `SL`
  - `SR`
  - `TL`
  - `TR`
  - 以及现有基础动作本身的 `combo_symbol`

这层的意义是把“输入 token”和“武器判招语义”之间先插入一层稳定边界。后面如果继续做：

- `WeaponPattern`
- `InputInterference`
- `Jump`
- 更复杂的相对动作识别

都可以优先建立在 `ActionTrace` 上，而不是继续直接读取 `ActionInstance` 或 `Game.gd` 的临时状态。

当前武器模型已经不再走 `WeaponTechniqueDef / WeaponComboResolver` 主链。保留 `ActionTrace` 的意义主要是：

- 记录动作真实执行结果
- 支撑调试和后续可能的输入语义分析

当前权威流程更简单：

1. `KeyProgram` 存可编辑基础 token
2. `DirectionalTechniqueResolver` 把 token 翻译成基础动作
3. 若 token 是攻击，则解析成当前武器声明的 `attack_action`
4. `ActionResolver` 直接解析该动作

### EffectPipeline

EffectPipeline 的扩展点是 `EffectModifierDef`：

- `modify_packets()`：在执行前修改、放大、复制或取消效果包。
- `react_to_event()`：对伤害、移动、击杀、击退等事件生成后续效果包。
- `priority`：控制修饰器处理顺序。
- `max_generation_depth` / `max_event_depth`：限制递归生成深度。

这套结构适合继续扩展遗物、诅咒、临时状态、武器词条和房间规则。

## 7. 视图与 UI

表现层主要在 `scripts/view/` 和 `scenes/ui/`：

- `BattleUI.gd`：战斗面板、行动队列、按键槽、敌人意图、奖励覆盖层、战斗结果。
- `BattleHud.gd`：玩家生命、SAN、房间、回合，以及玩家位置和朝向的统一状态框。
- `RunSidebar.gd`：右下角背包和菜单入口。`背包 / Tab` 会打开背包页并暂停当前场景；背包左侧显示 `QWER / ASDF / ZXCV` 三行四列的按键槽与永久增益，右侧显示未分配 token。菜单面板承接事件记录与调试状态。
- `BoardView.gd`：读取 `GameState` 渲染棋盘、危险格、预览移动格和攻击格。
- `UiKeySlot.gd` / `UiKeyToken.gd`：按键槽拖拽和 hover 预览。
- `SettingsMenu.gd`：分辨率、全屏、移动键重绑定。

UI 层的目标是展示和发信号，不直接拥有战斗规则。新增玩法规则时，优先放在 core、runtime 或 data 层，再由 view 层订阅状态刷新。

世界切片建筑与安全区 NPC 的当前落地方式：

- `WorldGenerator.gd` 依次执行 `fill_plain -> mountain_generation -> terrain_generation -> river_generation -> poi_placement -> obstacle_generation -> connectivity` 七段生成。
- `POIPlacementService.gd` 先挑 tavern / challenge / ruin 等单点 anchor，再把 anchor 交给建筑层。
- `BuildingPlacementService.gd` 会围绕 anchor 做本地搜索、扩展搜索和全局 reachable 搜索，挑出最合适的 pattern origin。
- `PatternStampService.gd` 把 ASCII pattern 压成 footprint，并写回 `interaction_cell`、`occupied_cells`、`poi:*`、`building:*`、`structure:*` 等标签。
- 当前 NPC 不参与全图随机撒点，而是由 tavern pattern 自带的 `npc_spawn_slots` 决定是否可生成，再从“玩家出生所在 tavern footprint”的 walkable cells 中挑选槽位落点，因此天然被限制在酒馆安全区。
- 运行态会记录：
  - `GameState.world_npc_positions`
  - `GameState.world_npc_display_names`
  - `GameState.tracked_world_npc_id`
  - `GameState.show_tracked_world_npc_hint`
  - `GameState.tracked_world_npc_relative_hint`
  这些字段用来支撑后续任务链、方向提示和追踪 UI。

## 8. Autoload 服务

`project.godot` 当前注册了这些 autoload：

- `SettingsService`：窗口分辨率和全屏设置。
- `SaveService`：统一存档入口，收集 provider 的 `get_save_data()`。
- `SceneService`：场景切换、重载和退出。
- `AudioService`：音乐、音效和音频 bus 控制。
- `AchievementService`：成就定义、事件、进度和存档。
- `RandomService`：可保存/恢复的随机数服务。
- `PlayerInputService`：移动键 InputMap 和重绑定。
- `CurseService`：run 内诅咒计数和规则。
- `EnemySpawnService`：敌人定义注册和按楼层生成候选。
- `EconomyService`：经济相关服务骨架。
- `DialogueManager`：第三方 Dialogue Manager。
- `DialogueService`：项目对话服务封装。

新增全局服务前先确认是否真的需要 autoload。若只是某个场景内部可控的逻辑，优先用子节点、Resource 或 RefCounted 组合。

## 9. 当前设计债务

这些不是立即要修的 bug，而是后续演进时应优先处理的结构点：

- `Game.gd` 仍承担较多装配职责，房间、路线、奖励表还写在常量里。
- `RunController.gd` 已存在，但当前主要 run 流程还在 `Game.gd` 中。
- `RoomDef.gd` 已作为未来数据驱动房间的脚手架，尚未接管 `ROOMS`。
- UI 已较完整，但 `ActorRoot` 与演员表现层后续可继续拆清楚。
- 对话、经济、诅咒等服务已有入口，玩法接入程度还较轻。

下一步丰富内容的推荐方向：

1. 先做两到三个真正改变站位思考的怪物模板，而不是继续只加数值更高的近战怪。
2. 先开放一到两个会改变编排思路的 token，例如 `SL / SR` 侧移和 `Dash` 冲刺，再围绕它们设计新武器与敌人。
3. 让出生 tavern 继续承担新 token 的第一次教学入口，但后续 token 主要改由地图拾取、宝箱、神龛或战后奖励发放。
4. 用“怪物压力类型 x token 解法类型”来组织后续内容，而不是把怪物扩展和键位扩展分成两条互不相干的线。
5. 具体到一轮小版本，最值得先落地的是：
   - 一个横向位移 token
   - 一个高承诺位移 token
   - 一只会逼玩家处理前后站位的敌人
   - 一只会逼玩家处理侧向位移或 timing 的敌人

下一步内容计划总结：

第一批最推荐做成一个完整组合：

1. 动作 / token：`SL / SR` 侧移。
2. 武器：匕首或双刀，一把明确依赖横向位移和贴身站位差的轻武器。
3. 怪物：直线压迫型敌人，一只会稳定威胁前方线、逼玩家横向挪位的怪。

第二批最推荐做成第二个完整组合：

1. 动作 / token：`Dash` 冲刺。
2. 武器：长枪深化版、枪盾或第二把重武器，强调前压、穿线或高承诺出手。
3. 怪物：延迟重击 / 区域压位型敌人，一只会蓄力封位、逼玩家承担位移决策的怪。

第三批再考虑更高复杂度内容：

1. 动作 / token：`Grab`、`ThrowPrep` 或背摔前置动作。
2. 武器：投技、背摔、重型控制武器。
3. 怪物：堵角重型怪、位移干扰怪，或更接近输入污染主题的精英敌人。

这三批的顺序不要轻易打乱，因为它们分别对应：

1. 先建立横向位移语义。
2. 再建立高承诺位移语义。
3. 最后再用更复杂的位移控制和输入干扰去放大系统主题。

推荐下一步拆分顺序：

1. 把 `ROOMS` 和 `MAP_NODES` 迁移为 Resource 数据或独立配置。
2. 让 `RunController` 接管路线推进、奖励构建和 run 持久状态。
3. 保持 `Game.gd` 为 composition root，只负责连接场景节点和协调刷新。
4. 为每个新增规则补一条 SmokeTest 或更小的 headless 测试。
