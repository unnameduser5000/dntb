# DNTB 系统设计文档

## 1. 项目定位

DNTB 当前是一个小体量、可快速迭代的 Godot 4 战术 roguelite 原型。它的核心不是传统“选择技能卡再行动”，而是把方向键抽象成可编程按键槽：玩家在安全节点或休息点调整 U/D/L/R 四个槽里的行动 token，进入战斗后按键槽锁定，按下对应键时依序执行槽内行动。

当前已经落地的体验包括：

- 8x8 网格房间、墙体、玩家、敌人和钥匙拾取。
- 营地/休息点可调整行动 token，战斗中锁定。
- 战斗内执行玩家行动链，然后敌人根据意图行动。
- 房间胜利后给奖励，奖励可增加行动、遗物或属性。
- 冲击盾武器、连续移动速度、撞击伤害和击退。
- EffectPipeline 支持伤害、移动、击退、事件反应和遗物式修饰器。
- 成就、设置、存档、输入绑定、随机服务等基础 autoload。

## 2. 运行入口

项目主场景在 `project.godot` 中配置为：

```text
res://scenes/game/App.tscn
```

入口结构分两层：

- `App.tscn`：应用外壳，负责主菜单、设置菜单、暂停菜单和游戏可见性切换。
- `Game.tscn`：核心玩法场景，负责棋盘、战斗 UI、回合控制、行动解析、敌人规划和 run 控制。

`App.tscn` 当前节点职责：

```text
App
├── AppBackground
├── Game
└── MenuLayer
    ├── MainMenu
    ├── SettingsMenu
    └── PauseMenu
```

`Game.tscn` 当前节点职责：

```text
Game
├── BoardView          # 棋盘显示
├── ActorRoot          # 演员表现根节点
├── CanvasLayer
│   └── BattleUI       # 战斗 HUD、行动槽、奖励和结算覆盖层
├── TurnController     # 回合流程
├── ActionResolver     # 行动规则解析
├── EnemyPlanner       # 敌人行动和威胁预览
└── RunController      # 早期 run 控制骨架
```

## 3. 目录分层

当前项目采用 Godot 常见的“场景、脚本、数据、资源”分离结构：

- `scenes/game/`：入口和核心玩法场景。
- `scenes/map/`：棋盘视图和棋盘格。
- `scenes/ui/`：菜单、HUD、UI 组件和主题。
- `scenes/actors/`：演员表现层场景。
- `scripts/core/`：核心服务、控制器和规则协调层。
- `scripts/data/`：Godot Resource 数据定义，例如 ActionDef、ActorDef、WeaponDef、EffectModifierDef。
- `scripts/runtime/`：一局战斗中的运行态对象，例如 ActorState、ActionInstance、EffectPacket。
- `scripts/view/`：场景脚本和 UI 表现层。
- `data/`：实际资源实例，包括角色、行动、遗物、武器和成就。
- `addons/dialogue_manager/`：第三方 Dialogue Manager 插件，保留其原许可证和结构。

## 4. 核心数据流

一次玩家按键行动的大致路径如下：

```text
键盘事件
→ PlayerInputService 判断 player_move_* 动作
→ Game._submit_key_chain(key_id)
→ ActionProgramController 把按键槽 token 转成 ActionInstance 队列
→ TurnController.submit_player_plan(plan)
→ ActionResolver.resolve(action, state)
→ EffectPipeline 处理 EffectPacket、Modifier 和事件反应
→ GameState / GridModel 被更新
→ EnemyPlanner 预览或生成敌人行动
→ Game 刷新 BoardView 和 BattleUI
```

这个路径里最重要的边界是：

- `Game.gd` 现在仍是玩法总装层，负责 run 进度、房间生成、奖励应用、UI 信号连接和存档 provider 注册。
- `ActionProgramController.gd` 是按键槽真实数据源，`Game.gd` 里的 `_key_slots` 和 `_loose_key_tokens` 只是兼容测试和调试的观察镜像。
- `TurnController.gd` 只管回合阶段和行动执行顺序，不直接写具体战斗规则。
- `ActionResolver.gd` 管移动、攻击、转向、防御、武器 hook、死亡和胜负判断。
- `EffectPipeline.gd` 管效果包的修饰、执行和事件反应，适合承载遗物、武器、状态效果等可组合规则。

## 5. 回合与 run 流程

当前 run 路线还写在 `scripts/view/Game.gd` 的 `MAP_NODES` 和 `ROOMS` 常量中。流程大致是：

```text
营地休息点
→ 第一战
→ 战后休息点
→ 第二战
→ 回复休息点
→ Boss 战
→ 通关或失败
```

战斗内循环：

1. `TurnController.start_battle()` 进入 planning。
2. 玩家通过 UI 队列或按键槽提交行动链。
3. `TurnController` 顺序执行玩家行动，并为连续同方向移动计算 chain speed 和 momentum。
4. `ActionResolver` 解析每个行动，必要时交给武器和 EffectPipeline。
5. 若战斗未结束，`EnemyPlanner` 生成敌人计划并执行。
6. 清理临时 flag，回合数加一，重新进入 planning。

## 6. 行动、武器和效果系统

### ActionDef

`ActionDef` 是行动资源，包含：

- `id`
- `display_name`
- `short_name`
- `kind`
- `range`
- `power`
- `direction_mode`

当前行动资源放在 `data/actions/`，例如移动、转向、攻击、等待、防御、突刺、横扫和方向移动 token。

### WeaponDef

`WeaponDef` 是武器基类，提供这些可选 hook：

- `resolve_move_collision()`
- `resolve_attack_hit()`
- `resolve_attack_miss()`
- `resolve_action_chain_finished()`

当前 `impact_shield.tres` 使用 `ImpactShieldDef`，设计重点是“移动撞击”：连续同方向移动会提高速度，撞上敌人时按速度造成伤害和击退。

### EffectPipeline

EffectPipeline 的扩展点是 `EffectModifierDef`：

- `modify_packets()`：在执行前修改、放大、复制或取消效果包。
- `react_to_event()`：对伤害、移动、击杀、击退等事件生成后续效果包。
- `priority`：控制修饰器处理顺序。
- `max_generation_depth` / `max_event_depth`：限制递归生成深度。

这套结构适合继续扩展遗物、诅咒、临时状态、武器词条和房间规则。

## 7. 视图与 UI

表现层主要在 `scripts/view/` 和 `scenes/ui/`：

- `BattleUI.gd`：战斗面板、行动队列、按键槽、奖励覆盖层、战斗结果。
- `BattleHud.gd` 和 `RunSidebar.gd`：玩家状态、敌人数量、回合、遗物列表等。
- `BoardView.gd`：读取 `GameState` 渲染棋盘、危险格、预览移动格和攻击格。
- `UiKeySlot.gd` / `UiKeyToken.gd`：按键槽拖拽和 hover 预览。
- `SettingsMenu.gd`：分辨率、全屏、移动键重绑定。

UI 层的目标是展示和发信号，不直接拥有战斗规则。新增玩法规则时，优先放在 core、runtime 或 data 层，再由 view 层订阅状态刷新。

## 8. Autoload 服务

`project.godot` 当前注册了这些 autoload：

- `SettingsService`：窗口分辨率和全屏设置。
- `SaveService`：统一存档入口，收集 provider 的 `get_save_data()`。
- `SceneService`：场景切换、重载和退出。
- `AudioService`：音乐、音效和音频 bus 控制。
- `AchievementService`：成就定义、事件、进度和存档。
- `RandomService`：可保存/恢复的随机数服务。
- `PlayerInputService`：移动键 InputMap 和重绑定。
- `CurseService`：run 内诅咒计数和规则。
- `EnemySpawnService`：敌人定义注册和按楼层生成候选。
- `EconomyService`：经济相关服务骨架。
- `DialogueManager`：第三方 Dialogue Manager。
- `DialogueService`：项目对话服务封装。

新增全局服务前先确认是否真的需要 autoload。若只是某个场景内部可控的逻辑，优先用子节点、Resource 或 RefCounted 组合。

## 9. 当前设计债务

这些不是立即要修的 bug，而是后续演进时应优先处理的结构点：

- `Game.gd` 仍承担较多装配职责，房间、路线、奖励表还写在常量里。
- `RunController.gd` 已存在，但当前主要 run 流程还在 `Game.gd` 中。
- `RoomDef.gd` 已作为未来数据驱动房间的脚手架，尚未接管 `ROOMS`。
- UI 已较完整，但 `ActorRoot` 与演员表现层后续可继续拆清楚。
- 对话、经济、诅咒等服务已有入口，玩法接入程度还较轻。

推荐下一步拆分顺序：

1. 把 `ROOMS` 和 `MAP_NODES` 迁移为 Resource 数据或独立配置。
2. 让 `RunController` 接管路线推进、奖励构建和 run 持久状态。
3. 保持 `Game.gd` 为 composition root，只负责连接场景节点和协调刷新。
4. 为每个新增规则补一条 SmokeTest 或更小的 headless 测试。
