# DNTB 开发协作指南

## 基本原则

- 先读真实入口：`project.godot`、`scenes/game/App.tscn`、`scenes/game/Game.tscn`。
- 保持改动聚焦。玩法规则、UI 表现、数据资源、测试验证尽量分清边界。
- 不提交 `.godot/` 缓存、导出包、本地日志、临时脚本缓存或个人 IDE 状态。
- Godot 生成并已纳入版本管理的 `.gd.uid`、`.import` 文件要跟随对应资源保留。
- 做玩法、架构、仓库流程上的明显变化时，同步更新 `DEVELOP_LOG.md`。

## 脚本边界

### core

`scripts/core/` 放控制器和服务。这里可以协调多个系统，但要避免混入具体 UI 布局。

适合放在 core 的内容：

- 回合流程和行动解析。
- 输入、存档、设置、随机、成就等服务。
- run 推进、敌人生成、奖励生成等跨场景逻辑。
- 只依赖数据和运行态，不依赖 Control 节点细节的规则。

### runtime

`scripts/runtime/` 放一局战斗中的短生命周期状态对象。它们通常是 `RefCounted`，不直接依赖场景树。

适合放在 runtime 的内容：

- `ActorState`
- `ActionInstance`
- `EffectPacket`
- `EffectEvent`
- `CombatContext`
- `TurnResult`

### data

`scripts/data/` 放 Godot Resource 类。它们定义可编辑、可复用的数据结构和规则扩展点。

适合放在 data 的内容：

- 行动定义。
- 角色和敌人定义。
- 武器定义。
- 遗物/修饰器定义。
- 成就、奖励、房间等数据定义。

### view

`scripts/view/` 放场景和 UI 表现脚本。它们读取状态、发出信号、刷新显示，不应成为新规则的主要承载处。

适合放在 view 的内容：

- 菜单和 HUD。
- 棋盘渲染。
- 拖拽、hover 预览、按钮响应。
- 场景根节点的组合和信号连接。

## 场景组织

当前场景拆分方向是合理的：

- `App.tscn` 管应用外壳。
- `Game.tscn` 管玩法组合根。
- UI 组件放在 `scenes/ui/components/`。
- 角色表现放在 `scenes/actors/`。
- 地图表现放在 `scenes/map/`。

新增场景时尽量遵守“一场景一职责”。如果一个场景节点数明显膨胀，或里面混入多个可独立测试的概念，就考虑拆成子场景。

## 数据资源约定

- 新行动优先新建 `data/actions/*.tres`，并在需要处注册到行动表。
- 新角色或敌人优先新建 `data/actors/*.tres`。
- 新遗物或规则修饰器优先继承/复用 `EffectModifierDef` 体系。
- 新武器优先继承 `WeaponDef`，通过 hook 接入移动碰撞、攻击命中、攻击落空和行动链结束。
- 资源里的中文展示文本保持 UTF-8，不要用系统默认编码另存。

## UI 改动约定

- UI 布局优先在 `.tscn` 和主题资源中维护，脚本只做必要的动态生成。
- 新按钮、卡片、状态行尽量复用 `scenes/ui/components/` 里的组件。
- 战斗规则不要写进 UI 脚本。UI 只负责把用户意图转成信号，再由 core 层处理。
- 背包页属于按键编排与查看 UI，不负责决定“当前是否允许编辑”；可编辑/锁定状态由 `Game.gd` 根据营地、休息点、酒馆休息区或战斗态统一下发。
- 背包页展示约定当前是：
  - 左侧：`QWER / ASDF / ZXCV` 三行四列键位槽与永久增益。
  - 右侧：未分配 token 池。
  - 同一键位内 token 从左到右顺序触发。
  - `Tab` 或屏幕按钮打开时会暂停游戏，关闭后恢复。
- 做分辨率相关改动时，至少检查 1280x720、1600x900、1920x1080 三档。

## 文档与日志

建议更新文档的场景：

- 系统边界改变，例如 run 逻辑从 `Game.gd` 移到 `RunController.gd`。
- 新增核心玩法，例如新武器 hook、新遗物事件、新房间数据格式。
- 验证命令或测试入口改变。
- 协作者需要知道的新资源命名规则。

`DEVELOP_LOG.md` 写“发生了什么、为什么、怎么验证”。设计文档写“现在系统长什么样、应该怎么扩展”。
